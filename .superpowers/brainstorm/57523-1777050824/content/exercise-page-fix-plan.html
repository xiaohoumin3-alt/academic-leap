步骤的定义、划分、以及展示必须清晰一致

一、步骤的核心定义（设计规范）

🎯 核心原则：步骤 = 数学解题的逻辑子目标

每个步骤应该是一个独立的、有意义的数学计算或判断，而不是人为拆分的输入框。

步骤设计规范表

模板 题目示例 步骤数 步骤设计
sqrt_concept √6 是否有意义？ 1步 直接判断二次根式是否有意义
pythagoras a=6, b=8, 求斜边c 2步 步骤1: 求 c² = a² + b² = 100
步骤2: 求 c = √c² = 10
quadrilateral_perimeter 正方形边长为4，求周长 2步 步骤1: 确认公式系数（周长/边长）
步骤2: 计算周长 = 4 × 4 = 16
trapezoid_property 等腰梯形，上底6，下底8，腰5 2步 步骤1: 确认公式（周长 = 上底 + 下底 + 2×腰）
步骤2: 计算周长 = 6 + 8 + 2×5 = 24

Bug 3: 步骤数显示错误（核心问题）

🔴 根因分析

用户看到的"三步"问题，根因可能是：

  1. 模板选择错误：知识点可能映射到了错误的模板
    "勾股定理" → pythagoras_folding（3步）而不是 pythagoras（2步）
  2. 模板实现与设计不一致：模板代码返回的步骤数与设计文档不符
  3. 硬编码 fallback：`currentQuestion?.steps?.length || 3`

用户实际体验：

"直角三角形 a=6, b=8，要求三步：第一步100，第二步10，第三步10"
❓ 为什么第三步答案和第二步一样？这不符合数学逻辑

修复方案：确保模板步骤数与设计一致

// lib/question-engine/templates/pythagoras.ts
buildSteps: (params): StepProtocolV2[] => {
  // 确保只有 2 步：第一步 c²，第二步 c
  return [
    { stepId: 's1', answerMode: AnswerMode.NUMBER, /* ... */ },
    { stepId: 's2', answerMode: AnswerMode.NUMBER, /* ... */ },
  ];
}

Bug 4: undefined 变量

问题示例

"等腰梯形上底为6，下底为8，腰长为undefined，求周长"

根因：模板中直接使用 `${params.leg}` 而没有默认值检查

// lib/question-engine/templates/chapter18/trapezoid_property.ts

// ❌ 错误：直接插值，无校验
title: `等腰梯形...腰长为${params.leg}，求周长`

// ✅ 正确：使用空值合并运算符
title: `等腰梯形...腰长为${params.leg ?? '未知'}，求周长`

// 或在 API 层添加参数校验

四、验证方法

✅ 正确场景

  • √6 有意义 → 显示"第 1 步 / 共 1 步"
