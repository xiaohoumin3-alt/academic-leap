## 一、背景与问题

### 1.1 当前系统问题

当前系统基于"步骤（Step）"概念，存在根本性问题：

| 问题 | 说明 |
|------|------|
| 步骤 ≠ 思维过程 | 现实中学生有多种解题路径，系统强加唯一顺序 |
| 公平性问题 | 学生A用10步，学生B用2步，答案都对，系统能给B扣分吗？ |
| 诊断矛盾 | 选择题只有1步，如何诊断学生卡在哪里？ |
| 步骤划分争议 | 什么该拆，什么不该拆，没有清晰标准 |

**核心矛盾**：系统试图用"线性结构"去约束"非线性思维"。

### 1.2 关键洞察

> 步骤是系统强加的结构，不是思维的本质。
> 真正的诊断能力来自：覆盖了多少认知节点，而不是走了多少步。

---

## 二、新范式：Question Graph

### 2.1 范式升级

```
旧范式（Step-based）:
一个题 → 多个步骤 → 强制顺序

新范式（Question Graph）:
一个知识点 → 多个子问题 → 弱依赖关系（DAG）
```

### 2.2 核心概念

**子问题（SubQuestion）**：一个"最小可判定的认知单元"

必须满足三个条件：
1. 学生必须输出（不是看/想）
2. 系统可以判对错
3. 对应一个独立认知动作

| 示例 | 是否子问题 | 原因 |
|------|-----------|------|
| "理解题意" | ❌ | 不可判定 |
| "判断是否直角三角形" | ✅ | yes/no 输出 |
| "计算 c²" | ✅ | 数值输出 |
| "计算 c" | ✅ | 数值输出 |

## 三、系统架构

### 3.1 数据结构

```typescript
// 知识点
interface KnowledgeUnit {
  id: string;
  name: string;
  chapter: string;
}

// 问题组（一个练习）
interface QuestionGroup {
  groupId: string;
  knowledge: KnowledgeUnit;
  difficulty: number;  // 0-5
  questions: SubQuestion[];
}

// 子问题
interface SubQuestion {
  questionId: string;
  type: AnswerMode;  // YES_NO | MULTIPLE_CHOICE | NUMBER | COORDINATE
  description: string;
  hint?: string;
  expectedAnswer: ExpectedAnswer;
  dependsOn?: string[];  // 依赖的其他问题ID
  difficultyWeight: number;  // 认知复杂度权重
}

// 答题记录
interface QuestionResult {
  questionId: string;
  isCorrect: boolean;
  duration: number;
  skipped: boolean;
}
```

### 3.2 依赖关系图（DAG）

```
示例：勾股定理应用题

Q1: 判断是否直角三角形 (boolean)
        ↓
Q2: 计算 c² (number)
        ↓
Q3: 计算 c (number)
        ↓
Q4: 应用到实际问题 (number)
```

**关键**：依赖关系只是"推荐顺序"，不强制执行。

## 四、核心设计决策

### 4.1 允许非线性答题

✅ **学生可以跳过前面问题，直接答后面**

**理由**：
- 公平性：高效解题的学生不应被惩罚
- 现实性：考试也是如此，没人要求按顺序做
- 诊断价值：跳题本身也是信息

### 4.2 动态补问机制

当学生直接答对高阶问题时：

```
学生: 直接答对 Q3 (计算 c)

系统推断:
  Q1: 可能掌握（但不确定）
  Q2: 可能掌握（但不确定）

系统动作: 动态补问
  "你答对了结果，我们验证一下基础：Q2 等于多少？"
```

**诊断从"流程内"变成"流程外补充"**。

### 4.3 诊断能力来源

❌ 旧系统：靠步骤数（1步 vs 3步）

✅ 新系统：靠问题覆盖范围

| 题型 | 覆盖的认知节点 | 诊断能力 |
|------|---------------|---------|
| 选择题（低难度） | Q2 | 基础计算 |
| 应用题（高难度） | Q1, Q2, Q3, Q4 | 完整推理链 |

**同一知识点，不同难度 = 不同覆盖范围**。

### 4.4 公式处理策略

| Level | 提示策略 | 理由 |
|-------|---------|------|
| 0 | "使用勾股定理" | 学生刚学，需要回忆 |
| 1 | "直角三角形边的关系" | 引导回忆 |
| 2+ | 无提示 | 学生应该掌握 |
| 3+ | 干扰信息 | 测试深度理解 |

**原则：提示是"认知触发"，不是"路径提供"**。

## 五、UI/UX 设计

### 5.1 问题卡片展示

```
┌─────────────────────────────────────┐
│  问题 2 / 4 (可跳转到其他问题)        │
├─────────────────────────────────────┤
│  在直角三角形中，a=3, b=4，求 c²    │
│                                      │
│  [ 输入框 ]                          │
│  [ 提交 ]   [ 跳过 ]   [ 下一题 ]    │
└─────────────────────────────────────┘
```

**关键变化**：
- ❌ 不显示"第X步"
- ✅ 显示"问题X/Y"，可自由跳转
- ✅ 显示依赖关系提示（可选）

### 5.2 进度展示

```
总进度: 2/4 问题完成

● ● ○ ○
Q1 Q2 Q3 Q4
↑   ↑
已完成 进行中
```

## 六、评估模型

### 6.1 掌握度计算

```typescript
interface MasteryProfile {
  // 每个认知节点的掌握情况
  nodes: {
    [questionId: number]: {
      mastered: boolean;
      confidence: number;  // 0-1
      lastAttempt: date;
    }
  };

  // 综合掌握度
  overall: number;  // 0-1

  // 推荐下一步
  recommended: string[];
}
```

### 6.2 诊断规则

| 场景 | 判断 |
|------|------|
| Q1正确, Q2正确, Q3正确 | 完全掌握 |
| Q1跳过, Q2正确, Q3正确 | 可能掌握（Q1未验证） |
| Q1错误, Q2跳过, Q3正确 | 部分掌握（基础不牢） |
| Q1跳过, Q2跳过, Q3正确 | 直接答题者（需补问验证） |

## 八、风险与缓解

| 风险 | 缓解措施 |
|------|---------|
| 数据迁移复杂 | 保留旧API，逐步迁移 |
| 学生不适应跳题 | 提供"推荐顺序"模式 |
| DAG 性能问题 | 依赖层级限制在3层内 |
| 诊断精度下降 | 增加动态补问触发条件 |
