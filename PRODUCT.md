## 1. 产品愿景 (Product Vision)

**让每个学生都能获得最适合自己当前能力的学习路径**

我们不是做一个"更智能的题库"，而是构建一个能真正理解学生能力、动态调整学习路径的自适应引擎。

---

## 2. 核心价值主张 (Value Proposition)

### 我们解决什么问题

| 传统问题 | 我们的方案 |
|---------|-----------|
| 题目太难 → 学生挫败放弃 | 动态降低难度，保持挑战可达成 |
| 题目太简单 → 学生无聊流失 | 识别能力边界，提供恰当挑战 |
| 千人一面 → 效率低下 | 个性化推荐，因材施教 |
| 不知道自己进步了多少 | 实时能力估计，可视化成长 |

### 为什么是我们

- **基于强化学习的自适应引擎**：不是规则引擎，是能从每个学生身上学习的系统
- **项目反应理论(IRT)能力估计**：量化学生能力，而非简单统计正确率
- **实时反馈循环**：每道题都在更新对学生能力的理解

### 主要用户

|---------|------|---------|
| **新手学生** | 基础薄弱，缺乏自信 | "题目太难，我做不出来" |

## 4. 产品目标 (Product Goals)

### 主目标

**LE (Learning Effectiveness) - 学生能力提升幅度**

```
LE = avg(post_ability - pre_ability)
目标：LE > 0.15 (至少提升15%)
```

这是我们存在的根本价值。如果推荐系统运行良好但学生没有进步，产品就失去了意义。

### 辅助目标

| 指标 | 定义 | 目的 |
|------|------|------|
| **留存率** | 7日/30日回访率 | 推荐合适 → 学生不放弃 |
| **完课率** | 完整练习session比例 | 学习体验流畅 |
| **推荐接受度** | 学生接受推荐题目的比例 | 信任指标 |

## 5. 核心KPI (Key Metrics)

### 技术KPI → 产品KPI 映射

| 技术指标 | 产品含义 | 目标值 |
|---------|---------|-------|
| **LE (Learning Effectiveness)** | 学生真的变强了 | > 15% |
| **CS (Convergence Stability)** | 系统推荐稳定，不乱跳 | > 85% |
| **DFI (Data Flow Integrity)** | 每次学习都可追溯 | > 99% |

### 业务KPI

| 指标 | 衡量方式 | 目的 |
|------|---------|------|
| 留存率 | D7/D30 回访 | 产品粘性 |
| 完课率 | 完成 session 的比例 | 体验流畅度 |
| NPS | 净推荐值 | 口碑传播 |

### KPI优先级

```
LE > CS > DFI > 留存率 > 完课率
```

**关键原则**：如果优化CS导致LE下降，是错误的优化方向。

## 6. 风险容忍度 (Risk Tolerance)

### 零容忍 (Zero Tolerance)

| 风险 | 后果 | 应对 |
|------|------|------|
| 系统失效导致学生流失 | 信任破坏，难以挽回 | 实时监控 + 自动降级 |
| 伪收敛（看似正常但无进步） | 学生浪费时间，意识到后流失 | 伪收敛检测 |
| 数据丢失 | 学习记录丢失，体验中断 | DFI监控 + 备份 |

### 可接受 (Acceptable)

| 风险 | 容忍边界 | 说明 |
|------|---------|------|
| 轻度推荐不准确 | 偶尔推荐不合适题目 | 学生可以理解 |
| 短期体验波动 | 噪声 < 10% | 系统鲁棒性范围 |
| 非核心功能异常 | 分析页面延迟 | 不影响核心学习流程 |

### 崩溃边界 (Collapse Boundaries)

| 指标 | 正常 | 警告 | 危险 | 崩溃 |
|------|------|------|------|------|
| 标签噪声率 | < 10% | 10-20% | 20-30% | > 30% |
| 反馈延迟 | < 5步 | 5-15步 | 15-30步 | > 30步 |
| LE | > 15% | 5-15% | 0-5% | < 0% |
| CS | > 85% | 70-85% | 50-70% | < 50% |

## 7. 数据假设 (Data Assumptions)

### 理想假设（当前系统基于此）

- 学生答题结果 = 学生真实能力体现
- 反馈即时可得
- 题目标签100%正确
- 数据分布稳定

### 现实情况（破坏性验证发现）

| 假设 | 现实 | 影响 |
|------|------|------|
| 即时反馈 | 学生可能离开、延迟答题 | credit assignment问题 |
| 稳定标签 | 题目可能出错、评分不一致 | 能力估计偏离 |
| 理性行为 | 学生可能随机点击、故意答错 | 推荐策略失效 |
| 固定分布 | 学生群体、题目难度会漂移 | 策略过时 |

### 容忍度策略

```
正常范围: 噪声 < 10% → 正常工作
警告范围: 噪声 10-20% → 继续工作 + 标记异常
危险范围: 噪声 20-30% → 触发降级
崩溃范围: 噪声 > 30% → 强制回滚
```

## 8. 权衡原则 (Trade-off Principles)

当目标冲突时的决策框架：

### 原则1: LE 优先于 CS

**场景**: 提高CS可能导致LE下降
**决策**: 优先LE，接受一定CS波动
**理由**: 宁可推荐有点波动但有效，不要稳定但无效

### 原则2: 信任保护优先于推荐准确性

**场景**: 不确定的推荐可能破坏信任
**决策**: 保守推荐，宁可略简单也不要过难
**理由**: 信任一旦破坏很难挽回

### 原则3: 检测优于隐藏

**场景**: 系统可能失效
**决策**: 必须能检测并报告失效状态
**理由**: 让学生知道"系统可能不准"比假装正常更好

### 原则4: 降级优于崩溃

**场景**: 检测到异常
**决策**: 自动切换到规则引擎等安全模式
**理由**: 简单但可靠优于复杂但崩溃

### 原则5: 透明优于黑盒

**场景**: 系统决策
**决策**: 向学生解释为什么推荐这道题
**理由**: 理解建立信任

## 8.5 游戏化系统 (Gamification System)

### 设计原则

游戏化系统是学力跃迁的激励机制核心，设计遵循以下原则：

1. **学习效果导向 (LE-Based)**
   - 所有奖励基于系统LE > 15%的成效
   - 成就与实际学习提升挂钩
   - 避免纯粹数量化的无效激励

2. **DFI优先保证**
   - 数据链完整性 >= 99%
   - 失败事件进入死信队列
   - 完整的事件追踪机制

3. **观察者模式**
   - 游戏化作为观察者，不阻塞学习流程
   - 异步处理保证核心性能
   - 错误隔离不影响主功能

4. **家长控制优先**
   - 安全第一，防止过度使用
   - 可配置的使用限制
   - 透明的使用报告

### 核心功能

#### 等级与经验系统

- **经验获得**: 基于练习完成度、准确率、连续学习
- **等级提升**: 经验值达到阈值自动升级
- **奖励机制**: 解锁新功能、成就、特殊权限

#### 成就系统

| 成就类别 | 示例 | 触发条件 |
|---------|------|----------|
| **学习成就** | 初试牛刀、知识大师 | 基于学习效果和能力提升 |
| **连续成就** | 三日连击、月冠军 | 基于坚持度和连续性 |
| **精度成就** | 神枪手、稳定优秀 | 基于答题准确率 |
| **探索成就** | 知识猎人、多元学习者 | 基于知识点覆盖和多样性 |

#### 排行榜系统

- **排行榜类型**: 每日榜、周榜、总榜
- **排名机制**: 基于经验值、学习效果、综合评分
- **隐私保护**: 家长可控制显示权限

#### 家长控制中心

| 控制项 | 默认值 | 选项 |
|--------|--------|------|
| **每日练习时长** | 60分钟 | 30-120分钟 |
| **每日题目数量** | 50题 | 30-100题 |
| **学习时间段** | 8:00-22:00 | 可配置 |
| **排行榜显示** | 显示 | 显示/隐藏 |

#### 监控与实验

- **健康监控**: 实时监控游戏化系统健康状态
- **A/B实验**: 持续优化激励机制效果
- **数据分析**: 学习行为分析和效果评估

### 技术实现

#### API架构

```typescript
// 核心事件处理
POST /api/gaming
// 处理练习完成、成就解锁等事件

// 排行榜查询
GET /api/gaming/leaderboard
// 获取各类型排行榜数据

// 家长控制
PATCH /api/gaming/parental/settings
// 更新家长控制设置
GET /api/gaming/parental/report
// 生成学习报告

// 监控系统
GET /api/gaming/monitoring/health
// 系统健康状态
GET /api/gaming/monitoring/experiment
// 实验状态查询
```

#### 数据模型

- **PlayerProfile**: 玩家等级、经验、属性
- **Streak**: 连续学习记录
- **Achievement**: 成就解锁记录
- **GamificationFailure**: 失败事件记录（DFI）
- **ParentalControl**: 家长设置配置

#### 关键指标

| 指标 | 目标值 | 监控方式 |
|------|--------|----------|
| **事件处理成功率** | > 99% | 实时监控 |
| **平均响应时间** | < 100ms | 性能监控 |
| **死信队列大小** | 0 | 定期检查 |
| **家长控制触发率** | 可配置 | 统计分析 |

### 设计约束

#### 性能约束

- 游戏化事件异步处理，不影响核心学习流程
- 使用Redis缓存排行榜和玩家数据
- 批量处理减少数据库压力

#### 安全约束

- 家长控制为最高优先级
- 所有数据操作需要验证权限
- 敏感信息加密存储

#### 扩展性约束

- 成就系统可动态配置
- 支持多种排行榜类型
- 实验系统支持A/B测试

## 9. 用户场景 (User Scenarios)

### 场景A: 新手学生

**初始状态**: 能力θ = -1.5，基础薄弱

**期望体验**:
- 不会因为题目太难而放弃
- 每次答题都有成就感
- 能看到自己的进步

**系统行为**:
- 推荐难度略低于当前能力
- 连续正确后逐步提升
- 显示进步可视化

**成功指标**: LE > 0, 7日留存

---

### 场景B: 进阶学生

**初始状态**: 能力θ = 0，中等水平

**期望体验**:
- 感到持续有挑战
- 能看到自己的能力提升
- 不会遇到无法解决的题目

**系统行为**:
- 推荐难度匹配当前能力
- 动态调整，保持最佳挑战区
- 及时反馈能力变化

**成功指标**: LE > 0.15, 完课率 > 80%

---

### 场景C: 高手学生

**初始状态**: 能力θ = +1.5，基础扎实

**期望体验**:
- 不会因为题目太简单而无聊
- 持续有新挑战
- 能探索更高难度内容

**系统行为**:
- 推荐难度略高于当前能力
- 快速提升到挑战区
- 提供拓展内容

**成功指标**: LE > 0.1, 持续使用

## 11. 产品边界 (Product Boundaries)

### 我们做什么

- ✅ 数学题目的个性化推荐
- ✅ 基于IRT的学生能力估计
- ✅ 自适应难度调整
- ✅ 学习效果分析与反馈

### 我们不做什么

- ❌ 内容创作（题目由AI生成，但内容质量不在产品核心）
- ❌ 社交功能（不是学习社区）
- ❌ 游戏化（不是靠积分徽章激励）
- ❌ 家长/教师端（专注学生体验）

- 完成Phase 3完整重构
